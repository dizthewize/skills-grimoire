# gh-issue Generate & Annotate — Design Spec

**Date:** 2026-04-15
**Status:** Approved
**Scope:** Upgrade gh-issue from a read/cache tool to an AI-powered task spec generator that writes structured acceptance criteria back to GitHub issues as comments

---

## Overview

`gh-issue` currently fetches and displays issue context, then writes a local cache file. The upgrade makes it a PM-equivalent tool: it reads a sparse GitHub issue, generates a full task spec (acceptance criteria, expected vs. actual behavior, edge cases, technical context), and posts it as a structured comment on the issue — visible to the whole team, not just the current Claude session.

Downstream skills (`fix-issue`, `develop-team`) check for the most recent gh-issue comment when context.json is absent or stale, so the generated spec persists beyond a single session.

---

## What Changes

| Area | Current | After |
|------|---------|-------|
| Generation | None — extracts what's already in the body | AI generates AC, expected vs. actual, edge cases, technical context |
| GitHub write | None | Posts structured comment via `gh issue comment` |
| Re-run behavior | Overwrites context.json | Posts new comment; old comments stay (history) |
| context.json | Primary data source for downstream skills | 30-min session cache only; downstream skills fall back to parsing latest gh-issue comment |
| .gitignore | Not mentioned | `.gh-issue/` added to `.gitignore` |

---

## Generated Comment Format

Posted as a GitHub comment via `gh issue comment {number} --body "..."`. Clearly labeled as AI-generated so reviewers know what came from the team vs. Claude.

```markdown
<!-- gh-issue:generated -->
## Task Spec

> Generated by /gh-issue on {date}. Update this issue if requirements change.

### Acceptance Criteria
- [ ] {criterion 1}
- [ ] {criterion 2}
- [ ] {criterion 3}

### Expected Behavior
{1-3 sentences describing what the system should do when working correctly}

### Actual Behavior
{1-2 sentences describing what is currently happening — derived from the issue description}

### Edge Cases
- {edge case 1}
- {edge case 2}

### Technical Context
{1-3 sentences about affected components, relevant code areas, or architectural notes — derived from labels, related issues, linked PRs, and issue body}

### Suggested Labels
{any labels not already applied that would help triage — or "None" if already well-labeled}
```

The `<!-- gh-issue:generated -->` HTML comment marker allows downstream skills to find the most recent generated spec via `gh issue view --comments`.

## Phase Changes to SKILL.md

Two new phases inserted into the existing pipeline. All existing phases (0-4) are otherwise unchanged.

### New Phase 2.5: Generate Task Spec

Inserted after Phase 2 (Enrich) and before Phase 3 (Display).

**Skip condition:** `output=display` was passed (user wants read-only, no writes).

**Input:** all data gathered in Phase 1 (issue body, title, labels, state, comments, author) and Phase 2 (linked PRs, related issues, project status).

**Generation prompt given to Claude:**

```
You are a product manager writing a task spec for a developer.

Issue: #{number} — {title}
Description: {body}
Labels: {labels}
Comments: {last 3 comments, truncated to 200 chars each}
Linked PRs: {titles and states}
Related issues: {titles and states}

Write:
1. Acceptance Criteria (3-6 checkboxes) — specific, testable, written from the user's perspective
2. Expected Behavior (1-3 sentences) — what the system should do when working correctly
3. Actual Behavior (1-2 sentences) — what is currently happening (from the issue description)
4. Edge Cases (2-4 bullets) — scenarios that should also be verified
5. Technical Context (1-3 sentences) — which components/areas are likely involved

Be concrete and specific. Do not pad. If the issue is already detailed, keep the spec tight.
```

Store the generated output as `GENERATED_SPEC` for Phase 2.6.

### New Phase 2.6: Post Comment to GitHub Issue

Inserted immediately after Phase 2.5.

**Skip condition:** `output=display` was passed.

```bash
gh issue comment {number} --repo {owner}/{repo} --body "$(cat <<'COMMENT'
<!-- gh-issue:generated -->
## Task Spec

> Generated by /gh-issue on {YYYY-MM-DD}. Update this issue if requirements change.

{GENERATED_SPEC formatted per comment template above}
COMMENT
)"
```

After posting, report:
```
Task spec posted to issue #{number}.
/fix-issue and /develop-team will use this spec automatically.
```

No check for existing gh-issue comments — always post fresh. Old generated comments stay as history.

## context.json Changes

Two additions to the existing schema:

```json
{
  "generatedSpec": {
    "acceptanceCriteria": ["...", "..."],
    "expectedBehavior": "...",
    "actualBehavior": "...",
    "edgeCases": ["...", "..."],
    "technicalContext": "...",
    "commentUrl": "https://github.com/owner/repo/issues/123#issuecomment-..."
  },
  "generatedAt": "2026-04-15T10:30:00Z"
}
```

`commentUrl` points to the posted comment so downstream skills can deep-link to it.

## .gitignore Update

The SKILL.md instructions section should tell users to add `.gh-issue/` to their project's `.gitignore`:

```
# Add to your project's .gitignore:
.gh-issue/
```

gh-issue itself should check for `.gitignore` in the working directory and append the line if `.gh-issue/` is not already present:

```bash
grep -q "\.gh-issue/" .gitignore 2>/dev/null || echo ".gh-issue/" >> .gitignore
```

Run this at the end of Phase 4 (Write Context File).

## Downstream Skill Behavior (fix-issue, develop-team)

When context.json is absent or stale (> 30 min old), downstream skills now have a fallback:

1. Fetch issue via `gh issue view {number} --comments --json comments`
2. Find the most recent comment with `<!-- gh-issue:generated -->` marker
3. Parse the structured sections from the comment body
4. Use as acceptance criteria context (same as context.json `generatedSpec` fields)

If no generated comment exists, fall back to extracting AC from the issue body directly (existing behavior).

---

## Parameters

One new parameter added to the existing parameter table:

| Parameter | Default | Description |
|-----------|---------|-------------|
| `generate` | `true` | Set `false` to skip AI generation and comment posting (pure read mode) |

`output=display` also suppresses generation (existing behavior, now explicitly documented).

---

## Out of Scope

- Editing the issue body — comments only
- Deleting or updating old gh-issue comments
- Generating specs for milestone mode (milestones aggregate issues; generation runs per-issue)
- Changes to fix-issue or develop-team SKILL.md beyond the fallback comment-reading logic
