How should the skill system be structured?

This shapes everything — naming, invocation, and how sub-skills connect

A

Single Orchestrator Skill

One entry point: /build-site. Handles the full lifecycle — questions, architecture, design, copy, code, deploy. Calls existing skills internally as phases. No separate sub-skills needed.

Pros

  • Simple to invoke
  • One place to configure
  • Clear phase ordering

Cons

  • Large, harder to maintain
  • Can't run individual phases standalone
B

Coordinator + Specialist Skills

A thin /build-site coordinator that delegates to purpose-built skills: /site-discovery, /site-design, /site-build, /site-deploy. Each skill is independently invocable.

Pros

  • Each skill reusable in isolation
  • Easier to test/improve one phase
  • Skills usable across other contexts

Cons

  • More files to author
  • State must be passed between skills
C

Pipeline with Resume Points

One /build-site skill with clearly named phases, each checkpoint-gated. State saved to .build-site/state.json. Can resume mid-pipeline, jump to a phase, or run a single phase with /build-site phase=design.

Pros

  • Resumable after interruptions
  • Flexible entry points
  • Single source of truth for state

Cons

  • More complex state management
  • Longer skill file
