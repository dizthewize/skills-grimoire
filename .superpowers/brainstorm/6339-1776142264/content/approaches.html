One key design decision remains — how site-builder works internally

The other skills (site-discovery, site-planner, etc.) are straightforward. The builder is the complex one.

A

Parallel Research → Phased Build (Recommended)

Mirrors the develop-team pattern. 5–6 parallel research agents first (stack explorer, component analyzer, Supabase schema designer, auth flow mapper, SEO structure mapper, copy integrator), then phased implementation with per-page commits and test generation.

Pros

  • Catches edge cases before writing code
  • Supabase schema and auth planned before any UI
  • Resumable via state.json

Cons

  • More upfront time before first file is written
  • More complex to implement
B

Sequential Phase-by-Phase Build

No parallel research agents. site-builder executes pages one by one: scaffold → homepage → core pages → auth pages → Supabase wiring → deploy prep. Each phase produces a commit.

Pros

  • Simpler to reason about
  • Faster to start producing files
  • Easier to implement and maintain

Cons

  • No holistic upfront planning
  • Schema decisions may need revisiting mid-build
C

Hybrid: Lightweight Research → Fast Build

A single architect agent does a 10-minute holistic review of the project context (not 5–6 parallel agents), produces a build plan, then sequential phased implementation begins. Middle ground between speed and planning depth.

Pros

  • Architectural coherence without full research overhead
  • Faster than full parallel approach
  • Simpler than Option A

Cons

  • One agent may miss domain-specific concerns
  • Less parallelism = slower than Option B
